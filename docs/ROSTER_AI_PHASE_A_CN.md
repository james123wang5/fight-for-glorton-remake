# 全角色 AI 阶段 A：数据协议

## 本阶段结论

阶段 A 只建立六个角色、四张地图共用的训练数据接口，不启动长时间训练，也不会替换
游戏里的 21/22 级模型。现有桃子 v5 仍保持原来的 294 维观察和 14 个目的动作，因此已经
认可的 22 级模型、真人录像和普通游戏不受影响。

新协议名为 `glorton-roster-purpose-v6`，输入共 480 维：

```text
前 294 维：冻结的桃子 v5 输入，逐值完全相同
后 186 维：角色、动作能力、地图与平台导航上下文
```

以后把桃子 v5 网络迁移到 v6 时，可以原样复制前 294 列权重，新增加的 186 列先置零。
这样新模型开始时与旧冠军输出一致，再逐步学习不同角色和地图，不需要丢掉现有成果。

## 186 维扩展数据

- 双方角色 one-hot、地图 one-hot。
- 双方静态动作能力：地面/空中手刀、上手刀、后投、护盾、闪避、普通特殊类型和
  上特殊类型。
- 双方角色物理参数：重量、水平速度、跳跃初速、重力、最大下落速度、身体宽高。
- 双方当前动作可用性：一段跳、二段跳、地面/空中手刀、上手刀、后投、地面/空中
  特殊、上特殊、护盾。
- 双方当前攻击的剩余帧数和剩余比例。
- 当前平台、对手平台、路线下一平台各 32 维 one-hot。
- 路线是否可达、距离、下一节点方向、移动平台标志、直线是否被墙挡住。
- 最近特殊弹体类型：铅笔、排泄物、垃圾和电球。

不同角色的特殊技不会再被统一解释成桃子的枪和火箭：SBL 的地面特殊是光束，Trash 和
Coffee 是抛物/陷阱，Trash 的上特殊是范围爆发；Aubergine 没有闪避动画，其合法能力位
也会明确标成不可用。

## 地图导航接口

导航图使用角色真实宽度、水平速度、跳跃初速、重力和地图当前平台位置生成。平台顶部
会换算为角色中心可安全落脚的区间；穿过平台顶部的实体墙会切断区间，所以 Mogadishu
不会再把建筑两侧误判为可以直接走过去。Rooftop 和 B52 的移动平台按当前相位重建。

当前图是“宽松候选图”，用于让策略知道该往哪一层走。阶段 B 会再用 25 ms 的真实物理
步进验证跳跃边，加入卡墙脱困、空跳和重复下落的奖励/惩罚测试。

## 检查命令

在项目根目录运行：

```bash
.venv-train/bin/python -m unittest training.tests.test_roster_contract -v

.venv-train/bin/python -m training.inspect_roster_contract \
  --output artifacts/training/roster_phase_a.json
```

第二条命令会输出六角色 × 四地图的 24 组平台节点、候选边、动态边和输入有限性检查，
同时列出未来并发任务各自的固定种子、模型目录和日志目录。

## 并发训练约束

阶段 B 的每个进程固定一个角色和一张地图。目录结构为：

```text
training/checkpoints/roster_v6/<角色>/<地图>/<run-id>/
training/logs/roster_v6/<角色>/<地图>/<run-id>/
```

每个任务会保存 `scenario.json`，并在运行时占用 `.train.lock`。相同 `run-id` 如果角色、
地图或种子不一致会直接拒绝，两个终端误跑同一任务也不会覆盖模型。

目前不要直接用旧的 `train_v5` 命令训练其他角色：旧环境仍把特殊技、课程和统计解释为
桃子专用逻辑。阶段 B 先接入每个角色的动作翻译、合法动作掩码、命中统计和专项奖励，
通过短程冒烟测试后，再给出可同时运行的五个正式命令。
